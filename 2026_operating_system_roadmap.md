# 2026 Operating System Roadmap

## Strategic Frame

**Time horizon:** April 23, 2026 → December 31, 2026  
**Core purpose:** Build a disciplined system that increases income, restores technical fluency, creates credible AI/ML proof, and converts effort into real opportunities in the UK.

This roadmap is not a motivational note. It is an operating document. It may be revised, but until revised, it governs decisions.

---

## The 3 Primary Goals

### Goal 1 — Economic Strength
Increase income, preserve runway, and aim aggressively toward a high savings outcome by year-end.

### Goal 2 — Career Transition
Move from café-based income toward a stronger technical trajectory through AI/ML, Python, data, or adjacent engineering roles.

### Goal 3 — Systems-Based Control
Replace vagueness, drift, and emotional decision-making with a repeatable operating system built on daily, weekly, and monthly execution.

---

## The 3 Fronts

## Front A — Cash Engine
**Purpose:** Generate money now, preserve stability, and create extra runway without derailing the long-term transition.

### What belongs to Front A
- Muffin Break shifts
- Freelance / gig work
- Data cleaning tasks
- Spreadsheet / CSV work
- Light Python tasks
- AI evaluation or structured annotation work
- Budgeting and savings tracking

### Front A success criteria
- Café shifts remain stable unless replaced by stronger income
- Gig proposals are sent weekly
- Extra income is pursued with discipline, not desperation
- Spending is tracked
- Savings are preserved or increased

### Front A failure points
- Chasing low-value gigs endlessly
- Letting freelance work consume the project/career transition
- Confusing movement with income
- Emotional spending after hard work

### Front A key rule
Cash work must support the transition, not replace it.

---

## Front B — Career Engine
**Purpose:** Rebuild fluency, produce technical proof, strengthen positioning, and create actual career capital.

### What belongs to Front B
- Daily Python practice
- Personal command-system project
- Book/document QA flagship project
- CV rewrites
- GitHub cleanup
- LinkedIn improvement
- Lee-related project work
- Narrow, practical technical learning tied to current needs

### Front B success criteria
- Code is touched at least 5 days per week
- A flagship project is shipped in versions
- CV / GitHub / LinkedIn become sharper and more credible
- Lee creates concrete assets: title, deliverables, evidence, or connections

### Front B failure points
- Starting too many projects
- Endless learning without outputs
- Perfectionism on the QA app
- Waiting to “feel ready” before building
- Letting Lee remain vague and unstructured

### Front B key rule
One flagship project, one fluency lane, one strategic collaboration lane. No chaos.

---

## Front C — Conversion Engine
**Purpose:** Convert work from Fronts A and B into actual outcomes: applications, interviews, referrals, clients, and sponsor-capable pathways.

### What belongs to Front C
- Job applications
- Sponsor-aware targeting
- Recruiter outreach
- LinkedIn networking
- Follow-ups
- Interview preparation
- Turning project work into CV bullets and portfolio signal
- Turning collaborations into references and proof

### Front C success criteria
- Consistent weekly application volume
- Outreach is deliberate, not random
- Opportunities are followed up
- Projects and work are converted into external signal
- Movement is visible in the pipeline

### Front C failure points
- Building privately without applying
- Relying emotionally on one opportunity
- Applying only to ideal titles
- Failing to track who was contacted and when

### Front C key rule
Hard work is not enough; it must be converted.

---

## Decision Criteria

Every opportunity, task, or idea should be judged through these four filters:

### 1. Money
Does this improve cash now or in the near future?

### 2. Fluency
Does this improve coding sharpness, technical confidence, or engineering rhythm?

### 3. Signal
Does this improve CV, GitHub, LinkedIn, portfolio, or interview story?

### 4. Leverage
Does this create useful contacts, referrals, credibility, or visa-capable job paths?

### Decision rule
- Scores on 0 filters → discard
- Scores on 1 filter → optional
- Scores on 2 filters → likely useful
- Scores on 3–4 filters → prioritize

## Strategic Principles

1. **Action before over-analysis**  
Clarity will come from contact with reality.

2. **One major system at a time**  
Finish Version 1 before expanding.

3. **Visible outputs matter more than internal effort**  
Every day must leave behind an artefact.

4. **Use versions, not perfection**  
Working → credible → presentable → engineered.

5. **Track reality, not mood**  
Measure progress weekly and monthly.

6. **Use ambition as a banner, not as a weapon against yourself**  
The stretch target is useful; self-destruction is not.

## Weekly Operating System

Each week has three rituals.

## 1. Sunday War Room
Set weekly targets for:
- shifts
- money / savings
- gig proposals
- coding days
- project milestone
- applications
- outreach
- one Lee or networking move

## 2. Midweek Correction
Check:
- Are applications behind?
- Has coding stalled?
- Is gig work consuming too much?
- Is the project drifting?
- Is Lee producing anything concrete?

## 3. End-of-Week Review
Measure:
- shifts completed
- money earned
- savings movement
- gig proposals sent
- applications sent
- coding sessions completed
- project milestone completed or not
- strongest front
- weakest front
- next week’s correction

## Daily Operating System

There are two day types.

## Shift Day
Goal: maintenance + forward motion.

### Minimum standard
- Front A: 10–15 min
- Front B: 45–60 min
- Front C: 15–20 min

### Strong version
- Front A: 20–30 min
- Front B: 60–90 min
- Front C: 20–30 min

---

## Non-Shift Day
Goal: decisive progress.

### Minimum standard
- Front A: 30–45 min
- Front B: 2 hours
- Front C: 45 min

### Strong version
- Front A: 45–60 min
- Front B: 3–4 hours
- Front C: 60–90 min

## Daily Checkpoints

## Morning Command Check
Answer:
- Is today a shift day or non-shift day?
- What is the one key task in Front A?
- What is the one key task in Front B?
- What is the one key task in Front C?
- What is the minimum acceptable day?

## End-of-Day Review
Answer:
- Did I move all three fronts?
- What concrete output was produced?
- What blocked me?
- What must resume tomorrow?

## Before-Sleep Lock-In
Answer:
- What is tomorrow’s first task?
- What is the first coding action tomorrow?
- What is the first admin / conversion action tomorrow?
